# 포케 딜레마 — 연구 설계 문서 (RESEARCH.md)

> 수업 도구(v1.0)를 **연구 도구**로 확장하기 위한 설계. 구현 지침은 §7 "research 모드 스펙" — 후임 개발 모델은
> 연구 설계(표본·조건 수)가 사용자에 의해 확정되기 전에는 구현에 착수하지 말 것. CLAUDE.md의 절대 규칙이 여기서도 우선한다.
> 상태: **설계 초안 (2026-07-10)** — 사용자 검토 대기.

## 1. 연구 배경과 차별점

- 이론적 뿌리: 트롤리 딜레마 → **Moral Machine** (Awad et al., 2018, *Nature*)이 자율주행이라는 옷을 입혀 대중화.
  본 연구는 그 옷을 한 번 더 갈아입힌 것 — **포켓몬**이라는 옷. 검증된 강제 이지선다 프레임워크를 차용하므로
  설계 정당성을 문헌 인용으로 확보한다.
- **차별점 (서론에 반드시 명시)**: Moral Machine의 복제가 아니다. 진짜 연구 질문은
  ① 죽음을 제거한 무해 딜레마에서도 동일한 판단 패턴이 나타나는가
  ② **윤리적으로 무관한 지각 속성(귀여움·강함·희귀함)이 분배 판단에 개입하는가**
  ③ 게이미피케이션 요소의 유무가 윤리 판단에 영향을 주는가.
- 무해 원칙(죽음 소재 금지)은 연구에서도 유지 — 미성년 대상 연구윤리(학교 IRB/생명윤리위) 관점에서 오히려 강점이다.

## 2. 연구 질문과 가설

| 가설 | 내용 | Moral Machine 대응 축 |
|---|---|---|
| **H1 귀여움 편향** | 윤리적 조건이 동일할 때, 지각된 귀여움이 높은 대상이 우선 선택된다 | (신규 — 본 연구의 독창 지점) |
| **H2 강함/유용성 편향** | 전투력·진화 단계가 높은 대상이 우선 선택된다 | 사회적 지위 편향과 평행 |
| **H3 희귀도 편향** | 레전더리/환상 포켓몬이 일반 포켓몬보다 우선 선택된다 | (신규) |
| **H4 수(數) 판단** | 다수(5) vs 소수(1)에서 다수 우선 경향 — **인간 대상(Awad et al.) 결과와 직접 대조** | 공리주의 축 |
| **H5 게임 요소 효과** | 포인트·도감 등 게이미피케이션 유무에 따라 판단 패턴이 달라진다 | (신규 — 조건 간 비교) |

- 타당도 프레임: "포켓몬 속성이 생명의 무게를 대변하는가"를 방어하지 않는다. 질문을 뒤집는다 —
  **"무관한 속성이 판단에 개입하는가"**. 개입의 존재 자체가 발견이며, 도덕심리학의 moral circle /
  mind perception 문헌과 접속된다.

## 3. 조작적 정의 — PokeAPI 필드 매핑 (이 프로젝트의 방법론적 독창성)

| 변수 | 조작적 정의 (PokeAPI) | 비고 |
|---|---|---|
| 귀여움 | `pokemon-species.is_baby`, `pokemon.height/weight`(소형) 후보 풀 → **파일럿 평정으로 타당화** | 학생 30명 내외, 7점 척도. API 지표와 평정의 상관 자체가 부록 거리 |
| 강함 | `pokemon.stats` base stat 총합, `evolution-chain` 진화 단계 | 상/하 풀을 총합 상·하위 사분위로 |
| 희귀도 | `pokemon-species.is_legendary`, `is_mythical` | 이지적 정의 — 조작 논란 없음 |
| 수 | 시나리오 내 개체 수 (1 vs 5) | 텍스트 조작 |
| 통제 | 부상 정도·상황 등 윤리적 조건은 시나리오 텍스트로 **양쪽 동일** 고정 | 속성 쌍만 변함 |

## 4. 실험 설계

- **설계**: 피험자 내(within-subject) 반복 측정. 딜레마 구조(치료 순서·구조 순서 등)를 고정하고
  속성 쌍 조건을 배치. 순서 효과는 **라틴 방격**으로 통제.
- **무선화**: ① 조건 제시 순서 ② **선택지 좌우 위치**(현 v1.0은 A 왼쪽 고정 — 연구 모드에서 필수 수정)
  ③ 포켓몬 개체(같은 조건 셀 안에서 풀 무선 추출 — 특정 개체 효과 상쇄).
- **프레이밍 제거**: 연구 모드에서는 선택지의 "(효율)" "(형평)" 라벨과 선택 후 성찰 문구(rf)를 숨긴다 — 응답 유도 방지.
- **H5 게임 요소**: 참가자를 게이미피케이션 조건(포인트·도감 수집 UI) vs 플레인 조건에 무선 배정 (between-subject).
- **표본**: 학급/학년 단위. 검정력 분석으로 조건 수 대비 최소 표본 산출 필요 (표본 설계는 사용자 확정 사항 — 미정).

## 5. 측정 (종속변수)

| DV | 수집 방법 | 용도 |
|---|---|---|
| 선택 (A/B) | 시행 로그 | 주 분석 — 조건별 선택률 |
| **반응 시간** | 제시~클릭 ms (구현 비용 0) | 판단 갈등 지표 |
| 확신도 (옵션) | 선택 후 4점 슬라이더 | 민감도 보조 |
| 윤리의식 변화 | **플랫폼 밖** 사전-사후 지필 검사 | 플랫폼 로그와 이원화 — 혼입 방지 |

## 6. 데이터 수집 아키텍처 (서버 없이)

1. 시행 단위 로그: `{참가자코드(익명), 조건, 딜레마id, 좌우배치, 선택, RT}` — **개인 식별 정보 저장 금지** (CLAUDE.md 규칙 4 유지, 참가자 코드는 교사가 오프라인 배부).
2. 종료 시 기계 판독 포맷(한 줄 인코딩) 출력 → **구글폼 사전 채움 URL 자동 생성**(원클릭 제출) + 복사 폴백.
3. `teacher.html` (ROADMAP v1.2와 통합): 제출 텍스트 일괄 붙여넣기 → 파싱 → CSV 다운로드 → 통계 패키지로.

## 7. research 모드 스펙 (후임 구현 지침 — 설계 확정 후 착수)

- `index.html`에 `?mode=research&cond=...` URL 파라미터로 진입 (수업 모드 기본값은 현행 유지 — **두 모드는 한 코드베이스**).
- research 모드에서: 이유 라벨·rf 숨김 / 좌우 무선화 / RT 로깅 / 성향 리포트 대신 "참여 완료 + 제출" 화면.
- 조건 정의는 `research-config.js`(신설)로 분리 — dilemmas.js처럼 데이터/엔진 분리 원칙 유지.
- 캐스팅: buildCast()를 확장해 조건이 요구하는 속성 풀(§3)에서 추출. PokeAPI 실패 시 연구 세션은 **중단 안내**(폴백 진행 금지 — 조작 무결성).
- 무해 검열·스키마 검증은 연구 딜레마에도 동일 적용.
